# ACP — a ponte com os agentes de código (AD-3 evoluído, ADR-0006, ADR-0008)

## Purpose & Scope
Toda a plumbing do Agent Client Protocol: subprocessos dos agentes, conexão JSON-RPC, handlers client-side (permission/fs/terminal/update) e sessões por-`(Agente, Worktree)`. Traduz "falar com o adapter ACP" em primitivas que o step `agent` dirige. **Nenhum comportamento de loop aqui** (AD-1) — só mecânica de protocolo.

## Entry Points & Contracts
- `openAgent(opts)` → `AgentHandle` (`agent.ts`): o Processo. Superfície: `ctx` (long-lived `ClientContext`, usado por `buildSession(cwd)`), `terminals`, `closed: Promise<void>`, `protocolVersion`, `agentInfo`, `cost`, `shutdown()` idempotente.
- `createAgentProcessPool(agentOptions, spawner, logger?, onReopen?)` → o pool **de processos** (`pool.ts`). **Spawn eager de todos os Agentes referenciados** no início do Run; falha de spawn derruba os já abertos (cleanup best-effort, engole erros) e re-lança → Run falha rápido. Métodos: `handle(agentName)`, `session(agentName, cwd)`, `peek`, `closeSession`, `closeAllSessions`, `shutdownAll()`. **Não existe `open()`/`closeAll()`.**
  - **Quais processos sobem**: os Agentes **referenciados** pelo Pipeline — o conjunto vem de `referencedAgents()` (em `../config/warnings.ts`), consumido por `../index.ts`. Perfil declarado no Registry e nunca usado **não spawna** (vira só warning).
  - Chave de Sessão: `${agent}::${worktree}` (`sessionKey`, aqui em `pool.ts`). Pool de **processos** = eager; pool de **sessões** = **lazy** (só abre quando um step `agent` roda de fato).
  - `env` do processo é **merge** (`{ ...process.env, ...envOverrides }`), não substituição.
- `createSessionPool(deps)` (`session.ts`) é **legado**: keyed **só por cwd** (`session(cwd)`/`peek`/`close`) e usado **apenas nos testes**. Produção usa `createAgentProcessPool`. Não construa código novo em cima dele.
- `capabilities.ts` (**novo, ADR-0008**): `AgentCapabilities` (shape pura: `modes`/`models`/`efforts` readonly + `configId`s) e `parseCapabilities(configOptions, fallbackModes?)` — parse puro (AD-6) dos `configOptions` do `session/new`, por categoria (`mode`/`model`/`thought_level`). Legacy `availableModes` é fallback para `mode` quando `configOptions` não anuncia a categoria. `category: "model_config"` (fast mode) ignorado intencionalmente (D35 — débito D-0010).
- `LoopySession` implementa `AgentSession` de `../types.ts`: `setMode` (**fail-closed contra Capabilities**, ver Pitfalls) / `setModel` / `setEffort` (best-effort + **warning visível** — ver Pitfalls) / `clear` (**reabre a sessão** — o `sessionId` muda) / `prompt` (resolve só com `stopReason`) / `readText` / `cancel` + métricas `drainUsage`/`readCost`. Expõe `capabilities` getter (`AgentCapabilities`) alimentado no `session/new` via `parseCapabilities`.
- `createClientApp(opts)` → registra TODOS os handlers **antes** de conectar (`client.ts`).

## Usage Patterns
- **Permissão por `kind`**: `session/request_permission` traz opções tagueadas (`allow_once`/`reject_once`/…). O resolver decide `allow`/`reject`/`cancel`, o handler casa o `optionId`. Default honra `acp.permissions.on_request`.
- **fs/terminal**: `fs/read_text_file`+`fs/write_text_file` via `FileSystemPort` (node fs); `terminal/*` via `TerminalManager` (spawn, merge stdout+stderr, respeita `outputByteLimit`). Anunciados em `initialize`.
- Classificação de turno (`classifyStopReason`): **só `end_turn` é sucesso**; `cancelled` é stop-signal nosso; `refusal`/`max_tokens`/`max_turn_requests` = falha do step.

## Anti-patterns
- Não escrever no stdout do subprocesso para nada além do transporte ndjson (stderr → logs).
- Não abrir mais de um Processo ACP **por Agente nomeado** — cada Agente referenciado tem exatamente um Processo (ADR-0006). Não trocar o cwd de uma sessão viva (imutável — abra outra sessão).
- Não ler o texto do turno via `readText()` cumulativo da SDK para veredito — use o buffer por-turno (ver Pitfalls).
- Não tratar `setMode` como best-effort igual a `setModel`/`setEffort`: a assimetria é deliberada (modo governa autonomia read-only × write).

## Dependencies & Edges
- Contratos `AgentSession`/`StopReason`: `../types.ts`. SDK: `@agentclientprotocol/sdk`.
- Consumido por `../index.ts` (`defaultRunLive` constrói o `createAgentProcessPool` a partir do `ResolvedAgents` + `referencedAgents`) e pelo step `../steps/agent.ts` (via `ctx.session`, injetado pelo `sessionProvider` do orquestrador).

## Patterns & Pitfalls
- **`clear()` REABRE a sessão** (não manda `/clear`): `dispose()` + `session/new`. Consequências que não se pode ignorar: o **`sessionId` muda**; `configOptions`/`availableModes` são re-parseados; **mode/model/effort guardados são re-aplicados** automaticamente; `costCarry` preserva a monotonicidade de `readCost()` entre reopens; e o callback **`onReopen`** obriga quem indexa por `sessionId` a se re-registrar — é o que remapeia `sessionToTask` em `../index.ts`. Sem isso, todo evento de TUI/tráfego pós-`clear` iria para o taskId errado.
- **`setMode` é fail-closed contra Capabilities (ADR-0008)**: valida o `modeId` contra `capabilities.modes` (descobertos por `configOptions[category="mode"]` — **não** `availableModes`, que o OpenCode deixa nulo) e **lança** com mensagem acionável quando não bate (o step `agent` re-lança nomeando step + agente). Modo é **Dialeto por-Agente** (claude: `acceptEdits`/`plan`; codex: `read-only`/`agent`/`agent-full-access`; opencode: `build`/`plan`) — um `mode:` do dialeto errado derruba o step na hora. A validação agora cobre **os três adapters** (inclusive OpenCode, que antes escapava).
- **Capabilities NÃO são estáticas — o response do `set_config_option` as refresca.** No OpenCode o `effort` são as **variants do model corrente**: ele só anuncia `thought_level` quando o model tem variants, e os valores **mudam com o model** (`glm-5.2` → `high, max`; `gpt-5` → `minimal…high`). Como o Step aplica `setModel` **antes** de `setEffort`, `setConfigOption` re-parseia os `configOptions` que vêm no response (`refreshCapabilities`) — sem isso o `effortConfigId` ficava `undefined` para sempre e **todo `setEffort` no OpenCode era pulado**. Corolário: a **Sondagem** (`probe-agent`) aplica o `model` antes de ler as capabilities, e o cache é keyed por **argv + model**.
- **`setModel`/`setEffort` (ADR-0006/ADR-0008, best-effort + warning visível)**: descobrem o `configId` por **categoria** (`model`; `thought_level`/reasoning) nos `capabilities` e emitem `session/set_config_option`. Capability ausente ou valor fora da lista ⇒ **ignora** (não chama `set_config_option`) e **emite `warning` visível** (`StoreEvent` `warning`, 14.º tipo — ADR-0008) em vez do antigo `logger.debug` silencioso. Não clampa: o motor não finge que `xhigh` do Codex "equivale" ao `max` do Claude. Erro do adapter ⇒ capturado, logado, engolido. Não existe fallback para `session/set_model`.
- **Buffer de turno (OQ3)**: `TurnTextBuffer` acumula `agent_message_chunk` por `sessionId`, resetado antes de cada prompt — é a fonte de verdade do texto do turno. Só chunks de texto entram; tool calls/plans/thoughts são streamados mas não bufferizados. É o que `parseVerdict` deve ler. (`readText()` tem fallback duplo — buffer OQ3 × texto drenado da SDK, escolhendo o maior.)
- **Timing OQ3**: o buffer é alimentado por handler de notificação *posterior* ao `prompt()` resolver. `runTurn` cruza uma macrotask (`flushSessionUpdates`/`setImmediate`) após drenar `readText()` da SDK para garantir buffer completo e não vazar chunk tardio para o próximo turno.
- **Conexão viva**: a SDK fecha uma conexão `connectWith` quando o callback resolve; por isso o callback resolve o handshake e depois `await gate.promise` — só aberto por `shutdown()`.
- `on_request: "policy"` (deny-patterns) ainda não implementado → resolve como `allow` (placeholder, config-change para ativar, fiel ao AD-1).
- **Captura de métricas (C-0005, best-effort)**: `drainUsage()` é acumulador **por-turno** drain-and-reset — soma `PromptResponse.usage` de todo `runTurn` e zera; `null` quando nenhum turno reportou. `readCost()` é snapshot **cumulativo da Sessão** (não zera; inclui o `costCarry` de reopens anteriores), lido do `CostBuffer` (`client.ts`) alimentado pelo stream `usage_update`. Ambos best-effort: `null` nunca falha turno/step (⇒ `n/d` a jusante). `usage_update.cost` é `@experimental`/UNSTABLE no SDK (cast frouxo). O cost assenta só **após** a barreira `flushSessionUpdates` do `runTurn`, por isso `readCost()` (chamado pelo orquestrador após `execute()`) lê valor estável.
- **onTraffic seam (C-0007, ADR-0005)**: `OpenAgentOptions.onTraffic?(entry, sessionId)` capta o Tráfego JSON-RPC **send+recv** (sends de `setMode`/`set_config_option`/`prompt`/`cancel` em `session.ts`; recvs de `session/update` + requests permission/fs/terminal em `client.ts`) e alimenta **dois** consumidores — o `TaskLogger.acp` e a store (`acp_traffic`). `onUpdate` continua entregando o `agent_message_chunk` (texto do Agente → Stream). Pura observação: sem os callbacks o boundary é **idêntico** (AD-1).
